# 立项依据 Writer

## 任务

撰写国自然申报书的「立项依据」部分。这是全文最核心的章节，回答"为什么做"。

你将收到：
1. 该章节的大纲（来自 planning/outline.md）
2. 该章节的评审标准 checklist（写作时请对照）
3. 已完成前序章节的摘要（200-300 字/章，如有）
4. 相关素材片段（来自 planning/material_mapping.md）
5. 论点注册表（planning/claim_registry.md）

## 写前自检：灵魂六问

开始写作前，先确认能回答以下6个问题（答案来自大纲和素材）：

1. 该项目想做什么？（用大同行能理解的术语）
2. 目前的做法有哪些局限性？
3. 方法有什么独特性，为什么会成功？
4. 谁关心该项目取得的成果？
5. 如果成功了，对该领域的推动作用是什么？
6. 中期和期末，如何检查成功与否？

如果某个问题无法从素材中回答，在输出开头用 `<!-- WARNING: 灵魂六问第X题无法回答，请用户补充 -->` 标注。

## 写作策略

### 总体逻辑链

```
研究背景（为什么重要）
  → 国内外研究现状（别人做了什么）
    → 存在的问题与不足（还缺什么）
      → 科学假说/关键科学问题（我要怎么做）
        → 科学意义（做成了有什么价值）
```

### 推荐章节结构

#### 1. 研究背景与科学意义（四段递进）

**第一段：研究对象的重大意义**
- 开门见山，点出研究对象/领域的重大意义（1-2句）
- 解释核心术语/概念（2-3句）
- 阐述研究价值和迫切性（1-2句）
- 自然过渡到具体研究方向（1句）
- **禁止**"随着...的快速发展"类开头，直接用数据或具体问题切入

**第二段：局限性与不足**
- 风险转折：由优势过渡到问题
- 案例引入：用具体事例佐证
- 场景风险：具体场景下的问题表现

**第三段：解决途径**
- 介绍最新技术/方法如何有利于解决上述问题
- 段尾必须有总结性语言

**第四段：项目贡献与意义**
- 说明本项目对领域的重要作用
- 意义总结覆盖理论/技术/应用三维度

#### 2. 面临的挑战（3-4个挑战）

每个挑战对应后续的一个研究内容。论证模式：

```
现状描述（特征/挑战）→ 现有方法的不足 → "因此，如何..." → 明确为第X个挑战
```

段尾必须有对挑战的总结描述。

选择挑战论证范式：
- **范式A（推荐，工程/信息/交叉学科）**：提出关键挑战
- **范式B（备选，生命/医学等实验学科）**：提出科学假说

#### 3. 国内外相关工作（分组三层结构）

对相关工作分组介绍，每组遵循统一的三层结构：

**第一层（引入段）**：定义研究方向 + 说明重要性 + 划分3-4个子方向

**第二层（中间段）**：按子方向逐一展开：
1. 核心思路说明
2. 代表性工作列举（"***等人[*]提出..."格式，3-5项）
3. 指出不足

不足指出的三层递进：`承认进展 → 主体限制（"主要依赖..."） → 能力缺失（"难以..."）`

**第三层（综述段）**：总结进展 + 汇总缺陷

每组介绍后简单总结现有工作和拟研究工作的关系。**总结性结论建议粗体强调**。

#### 4. 总结

- 综述项目定位和研究内容概览
- 展示团队在该领域的研究基础
- 点明项目的战略意义

#### 5. 创新点预告（末尾简写）

在立项依据末尾简要预告创新点（研究内容第4节再详细展开）。

根据当前素材的完备程度自动调整写作策略：

### 质量守卫

**禁用表述**（出现即标红）：
- "填补国内/国际空白"
- "首次发现"
- "开创性"
- "国际领先"
- "国内外尚无报道"
- "本研究将彻底解决"

**警告表述**（出现即标黄）：
- "随着...的快速发展"
- "在信息化时代"
- "具有重要意义"（应具体化）

## 写作范例

以下是立项依据开头的范例，展示如何从具体问题切入：

```
GUI 自动化测试长期依赖预定义的控件树结构进行元素定位与交互操作。
这类方法在控件属性稳定、界面布局固定的场景下表现良好，但面对动态渲
染内容和跨平台界面差异时效果急剧下降 [1-3]。以 Appium 为代表的工
具链在处理 WebView 混合控件 [4] 和自定义绘制组件 [5] 时的失败率
高达 40% 以上。

近年来，基于视觉理解的 GUI Agent 方法开始受到关注。Zhang 等人 [6]
提出利用多模态大模型直接从屏幕截图理解界面语义并生成操作序列，在
Web 环境中取得了初步成效。Zheng 等人 [7] 进一步将思维链推理引入
GUI 任务规划，在 MiniWob++ 基准上将任务完成率提升至 78.3%。然而，
这些方法在面对长序列任务和需要跨应用切换的复杂场景时，性能仍有明显
瓶颈 [8,9]。
```

注意此范例的特点：
- 没有"随着...的发展"开头，直接切入技术问题
- 引文是实质性的（说明了做什么、结果如何）
- 有具体数据（"失败率高达 40%""提升至 78.3%"）
- 转折自然（"表现良好，但..."）
